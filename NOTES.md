   ## 2026-07-13  - WSL2, Ubuntu и основы Linux
   - Проверил состояние: wsl --status, wsl -l -v → WSL не установлен. Принцип «сначала факты, потом действия» (чтобы --install ничего не затёр).

   - wsl --install (PowerShell от администратора — аналог sudo) → поставил WSL2 + Ubuntu 26.04 LTS, перезагрузка, создал UNIX-пользователя richard + пароль (при вводе пароля в Linux экран пустой — норма).

   - Проверка: wsl -l -v → Ubuntu Running VERSION 2.
   - WSL2 vs WSL1: WSL2 = настоящее ядро Linux в лёгкой ВМ (100% совместимость системных вызовов); WSL1 = слой-переводчик syscall'ов. Docker требует фич ядра (namespaces, cgroups) → живёт на WSL2.

   - Вошёл в Ubuntu (pwd → /home/richard), sudo apt update — обновил каталог пакетов (ничего не ставит). sudo = права root (как UAC), apt = менеджер пакетов (сам разруливает зависимости).

   - sudo apt upgrade — применил обновления (67 пакетов). Узнал: phased updates (поэтапная раскатка — 2 пакета «not upgrading due to phasing» это норма), 48 LTS security updates (подтвердил LTS), warning ≠ error, часовой пояс подхватился из Windows.

   - apt list --upgradable, lsb_release -a → Ubuntu 26.04 LTS (resolute). Заодно попал в пейджер less (полный экран с ~), выход — q (survival: q для less, :q! для vim, Ctrl+X для nano).

   - /mnt/c vs родная Linux-ФС: /mnt/c — диск Windows изнутри Linux, доступ через протокол 9P (как «по сети») → медленно; код проекта держим в ~/ (ext4, быстро).

   ## 2026-07-14  - git, установка Go, первая программа

   - mkdir ~/projects, git --version → 2.53.0 (git уже стоял).

   - git config --global user.name/user.email (конфиг в ~/.gitconfig, свой в каждом окружении); клонировал lab-storage в ~/projects/lab-storage (публичный, HTTPS, без токена для clone).

   - git vs GitHub: git — инструмент версий, работает локально; GitHub — хостинг/удалённое хранилище. Папка .git = «мозг» репо (вся история); git init делает репозиторий локально без GitHub (git распределённый — каждый клон = полная история).

   - Установка Go — официальный tarball, не apt (дистрибутивные версии отстают): uname -m → x86_64 (нужен amd64); нашёл go1.26.5.linux-amd64.tar.gz на go.dev + SHA256.
   - curl -LO <url> — скачал; sha256sum — сверил контрольную сумму с опубликованной (защита от порчи/подмены; хэш = односторонний отпечаток, не шифрование).

   - Распаковка: sudo rm -rf /usr/local/go + sudo tar -C /usr/local -xzf go1.26.5...tar.gz → /usr/local/go. Флаги tar (-C, -x, -z, -f); /usr/local по FHS = софт, ставящийся вручную; sudo нужен, т.к. каталог принадлежит root (а ~ — мне).

   - PATH: echo $PATH (каталога Go нет; WSL подмешивает Windows-PATH); go version → not found; /usr/local/go/bin/go version работает (полный путь минует поиск по PATH). PATH = список каталогов, где шелл ищет команды.
   - Прописал навсегда: nano ~/.bashrc + export PATH=$PATH:/usr/local/go/bin (обязательно $PATH: в начале, иначе затрёшь весь PATH); source ~/.bashrc применил в текущем шелле.

   - source vs запуск скрипта: обычный запуск = дочерний процесс (свой env, изменения умирают с ним); source = выполняется в текущем процессе. Окружение течёт родитель → ребёнок, не наоборот.
   - go version + which go → работает.

   - Первая программа (~/projects/hello): go mod init hello; main.go (package main, func main, fmt.Println); go run . → Hello, Go!.

   - go build → постоянный бинарник hello (~2.4 МБ); запуск ./hello (нужен ./, т.к. текущей папки нет в PATH). Бинарник статически слинкован / самодостаточен (рантайм Go внутри; на целевой машине Go не нужен — в отличие от Java+JVM).
   
   - Кросс-компиляция: GOOS=windows GOARCH=amd64 go build -o hello.exe — Windows-бинарник с Linux-машины; file hello hello.exe → ELF … Linux vs PE32+ … MS Windows. Самодостаточность ≠ независимость от ОС/архитектуры; второе решается кросс-компиляцией.
